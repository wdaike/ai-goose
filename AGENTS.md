# AGENTS Instructions

iCodex is a frontend for OpenAI's Codex app-server. It ships as an Electron
desktop app and can also run in a plain browser through a thin Node host. There
is no Rust code and no backend of our own.

## Architecture

The renderer speaks the app-server v2 protocol directly through a four-verb
`window.codex` faĂ§ade (`request`/`notify`/`respond`/`onEvent`). Everything below
that faĂ§ade is transport, and all of it lives in one place â€”
`src/codex/codexProcess.ts` â€” which spawns `codex app-server` (the
official binary, from `PATH` or `GOOSE_CODEX_BIN`) and handles the stdio JSONL
JSON-RPC, request correlation, and forwarding of server-initiated messages via
an injected `onServerMessage` sink. Two thin adapters back `window.codex` with
that core:

- Desktop: `src/codex/bridge.ts` runs `CodexProcess` in the Electron main
  process and fans messages to the renderer over IPC.
- Web: `src/web/host.ts` runs `CodexProcess` in a single-port Express host that
  serves the renderer and carries the protocol over HTTP POST (request/notify/
  respond) + Server-Sent Events (codexâ†’client). The browser side is
  `src/codex/browserCodex.ts`, installed as `window.codex` by
  `src/browserHost.ts`.

Add a new transport by supplying an `onServerMessage` sink to `CodexProcess`;
never fork the protocol handling itself.

Codex owns everything backend-shaped: inference, tool execution, the MCP
client, prompt assembly (including `AGENTS.md` discovery), conversation
storage (`thread/*`), skills, config (`config/read`, `config/batchWrite`),
model catalog (`model/list`), and accounts. Don't reintroduce a parallel
implementation of anything Codex already provides â€” route to it instead.

Key renderer modules:

- `src/codex/client.ts` â€” typed request faĂ§ade over `window.codex` (thread/turn,
  model/config/account, skills, and `fs/*` used by the browser file host)
- `src/codex/protocol/` â€” TS types generated by `codex app-server generate-ts`
- `src/codex/engine/controller.ts` â€” chat engine: drives `thread/start`,
  `turn/start`, `turn/steer`, `turn/interrupt`, approvals; implements the
  `AcpChatSessionController` interface the UI consumes
- `src/codex/engine/mapItems.ts` â€” maps Codex thread items to the UI's
  `Message` model (agentMessageâ†’text, reasoningâ†’thinking,
  commandExecution/mcpToolCall/fileChangeâ†’toolRequest/toolResponse)
- `src/acp/` â€” legacy data-layer modules, now codex-backed or stubbed; the
  chat store (`chatSessionStore.ts`) and permission plumbing are still live

## Commands

```bash
pnpm install
pnpm run start-gui   # desktop app (needs `codex` on PATH)
pnpm run start-web   # browser app on http://127.0.0.1:5173
pnpm run typecheck
pnpm test
npx eslint src --ext .ts,.tsx
```

`./start-desktop.sh` / `./start-web.sh` at the repo root wrap those with a
`codex`-on-PATH check and first-run install. The web host serves without a token
on loopback; set `GOOSE_WEB_TOKEN` to require one. `pnpm run build:web` +
`pnpm run serve-web` runs the production static form of the web app.

To regenerate protocol types after upgrading codex:

```bash
codex app-server generate-ts --out src/codex/protocol
```

## Structure

```
src/main.ts        # Electron main; runs the codex bridge
src/codex/         # codexProcess (transport core), bridge (IPC adapter),
                   # browserCodex (WS/SSE window.codex), client, protocol,
                   # engine (chat controller + item mapping)
src/web/           # single-port Node host for the browser build
src/browserHost.ts # installs window.electron + window.codex in the browser
src/acp/           # legacy seams, codex-backed or stubbed
src/components/    # UI (unchanged iCodex look & feel)
```

## Rules

- Codex: All backend behavior belongs to `codex app-server` â€” extend
  `src/codex/engine/` rather than reimplementing anything Codex provides
- Open source first: Before building custom functionality, evaluate mature
  open-source solutions and the project's existing dependencies. Reuse them
  when they satisfy the requirements, and only implement the uncovered gaps.
- MCP: servers are configured in codex `config.toml` (`mcp_servers.*`) via
  `config/batchWrite`; status/tools via `mcpServerStatus/list`
- UI Desktop: Use local `src/types/*` types. Do not import generated OpenAPI
  types/client code from `src/api`
- UI prefs that have no codex home live in `localStorage` (`src/acp/config.ts`)

## Code Quality

- Comments: Write self-documenting code - prefer clear names over comments
- Comments: Never add comments that restate what code does
- Comments: Only comment for complex algorithms, non-obvious business logic, or "why" not "what"
- Simplicity: Don't make things optional that don't need to be
- Simplicity: Booleans should default to false, not be optional
- Simplicity: Avoid overly defensive code

## Never

- Never: Recreate `src/api` or add `@hey-api/openapi-ts`
- Never: Hand-edit files under `src/codex/protocol/` â€” they are generated
- Never: Reintroduce an iCodex-owned backend (ACP server, message store, MCP
  client). The web host (`src/web/host.ts`) is not one â€” it only spawns
  `codex app-server` and transports its protocol; keep it that thin.

## Entry Points

- Web host (HTTP + SSE): src/web/host.ts
